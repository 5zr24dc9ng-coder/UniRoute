## 2026-07-07：プレミアム判定はlocalStorageに永続化しない

### 決定
`?admin=true`によるプレミアム解放（`isPremium`）は、URLパラメータがある間のセッションのみ有効とし、localStorageには一切保存しない。

### 理由
以前のバージョンでは解放状態をlocalStorageに保存していたため、`?admin=true`を一度でも踏んだ端末（開発者自身のブラウザ含む）で、以後ずっとプレミアムが無料で見え続けるバグがあった。誰でも推測できるURLパラメータで永久に課金を回避できてしまう状態は事業上致命的なため、絶対に同じ設計に戻さないという制約として明文化する。

### 却下
localStorageでのキャッシュによる高速化・UX向上案 → プレミアム判定に関しては一切採用しない。

## 2026-07-07：Supabase `users`テーブルの列単位権限管理

### 決定
```sql
REVOKE UPDATE ON public.users FROM authenticated;
GRANT UPDATE (simulation_state, task_state) ON public.users TO authenticated;
```
`is_premium`・`stripe_customer_id`はservice roleキーを使うStripe webhookからしか更新できないようにした。

### 理由
セキュリティ監査の過程で、SupabaseのRLSが「自分の行のみ」という行単位の制御であり列単位の制御ではないことが判明。ログイン中のユーザーがブラウザの開発者ツールから直接`is_premium: true`を書き換えられる状態だった（実際にNetworkタブでの検証により再現・修正確認済み）。

### 却下
RLSのポリシー文だけで列アクセスを制限する案 → RLSは行単位の制御であり列単位の防御には使えないため、GRANT/REVOKEでの列単位管理を採用。

## 2026-07-07：Clerk-Supabase連携をThird-Party Auth方式に移行

### 決定
`useSupabase.ts`の`session.getToken({ template: "supabase" })`を`session.getToken()`（テンプレート指定なし）に変更。Clerk側のSupabase統合設定・Supabase側のThird-Party Auth設定を有効化。

### 理由
旧JWTテンプレート方式は2025年4月1日付で公式に非推奨。加えて、このアプリではそもそも指定していたテンプレート自体がClerk側に存在せず、クラウド同期機能がリリース以来実質的に一度も正常動作していなかった可能性が高いことが判明した。

### 却下
旧方式のままテンプレートだけ作成して延命する案 → 非推奨方式に依存し続けるリスクがあるため、新方式へ完全移行する方を選んだ。

## 2026-07-09：追加セキュリティ検証（IDOR・Webhook偽装・クライアント側シークレット漏洩・XSS・DDoS）

### 決定
5項目を再検証し、いずれも追加の実装なしで安全と確認した。

### 理由
- IDOR：`users`テーブルのRLSが`auth.jwt() ->> 'sub'`とid列を照合しており、クライアントが任意のidを指定しても他人の行にはアクセスできない
- Webhook偽装・リプレイ：署名検証済み、かつ処理内容が冪等なため実害がない
- クライアント側シークレット：`VITE_`プレフィックスの3つ（Clerk publishable key / Supabase URL / anon key）のみが公開されており、いずれも公開前提の値
- XSS：`dangerouslySetInnerHTML`・`innerHTML`・`eval`の使用ゼロ
- DDoS（コスト型含む）：唯一の懸念だった`api/fx.ts`は`Cache-Control: s-maxage=86400`のCDNキャッシュで連打の実害を実質防げている

### 却下
現時点でのレート制限（Upstash Redis等）の実装 → 導入方法自体は整理したが、現状の規模では緊急性が低いため見送り、学習目的の資料として`ARCHITECTURE.md`等に留める。

## 2026-07-10：外部監査（Fable）指摘4件への対応

### 決定
以下4点を修正した。

1. `.env`にClerkの本番シークレット（`sk_live_`）とテスト用の`pk_test_`が混在していた問題 → Clerkダッシュボードで`sk_live_`をローテーションし、ローカル`.env`には本番鍵を置かず、Clerkのテストインスタンスの鍵に統一する運用に変更（本番鍵はVercelの環境変数のみに置く）
2. `api/fx.ts`が環境変数名一覧・総数を`_diag`としてレスポンスに含めていた → レスポンスから削除し、ログ（`console.error`）にのみ残すよう修正
3. `create-checkout-session.ts`・Stripe webhook・Clerk webhookが、エラー時に`detail: err.message`をレスポンスに含めていた → 全箇所で`detail`を削除し、詳細はログにのみ残すよう修正
4. `vercel.json`にセキュリティヘッダー（`X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`, `Content-Security-Policy: frame-ancestors 'none'`）が無かった → 追加

### 理由
Fable（Claudeの別モデル）によるセキュリティレビューで発見。`.env`の鍵混在は、本番シークレットがローカル開発環境に存在すること自体がリスクであり、かつAIツールにファイル内容を読ませた時点で一種の第三者露出が発生しうるため、ローテーションを最優先で実施した。他3点は攻撃者に渡す情報を最小化する「防御の多層化」目的。

### 却下
CSPを`script-src`等まで厳格化する案 → Clerk/Stripe/Supabaseへの外部接続が多く、厳格化すると壊れるリスクが高いため、まずは`frame-ancestors 'none'`のみの最小構成で導入し、段階的に強化する方針にした。

## 2026-07-20：トビタテ！留学JAPANの奨学金プリセット額を公式値に更新

### 決定
`ScholarshipOffset.tsx`の`tobitate`プリセットの`lumpSum`を150,000円→510,000円に修正。`note`を「留学準備金15万〜25万円」から「留学準備金21万〜35万円（地域による）＋授業料支援30万円」に修正。代表値は準備金の下限21万円＋授業料支援30万円＝51万円を採用（月額支給の代表値は引き続き中央値12万円のロジックを踏襲）。

### 理由
2026年度（第18期）公式募集要項との突合で判明。旧データは公式より低い金額になっており、ツールが奨学金の実質的なカバー額を過小評価していた（ユーザー側の記事執筆時の一次情報確認で発覚・報告を受けて修正）。授業料支援は準備金と性質が異なる別枠の一時金だが、`totalScholarship`は控除額の合計としてのみ使われる（`realCost = 総額 - totalScholarship`）ため、単一の`lumpSum`に合算し`note`で内訳を明記する形で対応した。

### 却下
`Preset`型に内訳フィールドを追加し、UIに「準備金」「授業料支援」を別々の行で表示する案 → 計算結果に影響しないこと、既存の`note`欄で内訳表示の役割を十分果たせることから、型変更・UI変更を伴う対応は見送った。

## 2026-07-21：カナダの学生ビザ承認率を修正（91%→41%）、visaDifficultyを1(低)→3(高)に変更

### 決定
`src/constants/countries.ts`と`scripts/generate-visa-timing-pages.mjs`（静的SEOページ生成用の複製データ）の両方で、CAの`approvalRate`を91→41、`visaDifficulty`を1→3、`visaLabel`を「低」→「高」に修正。UK/US/AUの値は変更しない。

### 理由
Kaiによる一次情報の検証で判明。IRCC（カナダ移民局）公式統計によると、留学生数上限（Cap）導入前の2023年は全体承認率約59%、Cap導入初年度の2024年は49%（一部時期46%）、直近の2025年1〜9月は41%（ICEF Monitor記事によれば年間ベースで35.7%まで低下との報道もあり）。91%という旧数値は、Cap導入前・特定条件下のデータだった可能性が高く、現状の実態（コミュニティ全体で4割前後まで低下）と大きく乖離していた。

`visaDifficulty`の変更は、Kaiの指示（「相対的に判断して」）を受けて4カ国のデータを比較した結果。承認率はUK89%・AU85%・US82%・CA41%でCAが唯一大きく突出して低く、審査期間も112日でUK15日・AU18日・US30日と比べて最長。両指標でCAが4カ国中最も厳しい状態になったため、最高難易度である3（高）に変更した（UK/US/AUの相対順位・数値は今回のスコープ外につき据え置き）。

出典：
- IRCC「International Students」2025年12月4日付委員会資料
- IRCC「International Student Program at a Glance」2024年11月4日付委員会資料
- ICEF Monitor「New international student permit approvals for Canada fell below COVID levels in 2025」

### 却下
UK/AU/USの`visaDifficulty`も含めた4カ国全体の再ランキング → 今回の指摘・裏取りはカナダの数値のみが対象のため、他国の再評価は別タスクとして切り離した。
